# Mail Sorting System

Хакатон по Технологиям программирования, УЦП, июнь 2026.

Мы сделали небольшую систему для сортировки писем IT-поддержки.
Идея простая: есть папка inbox/ с письмами, программа читает их, определяет категорию и копирует в нужную папку внутри processed/.

## Как работает

На входе папка `inbox/` со 109 письмами — обычный текст, json, попадаются бинарные файлы и пустые. Мы всё это читаем, вытаскиваем тему/отправителя/текст, определяем категорию и копируем в `processed/<категория>/`. Параллельно пишем лог и считаем статистику.

### Категории

Сначала мы пролистали все письма из выданного набора и выписали что там вообще встречается. Получилась такая картина: куча пользовательских запросов, инциденты и сбои, кадровые вопросы, автоуведомления мониторинга, спам/фишинг, счета и бухгалтерия, инфо-рассылки. Выделили 8 осмысленных категорий плюс одна для всего остального:

- `incident` — ошибка 500, сбой, не отвечает, критический инцидент, не могу войти
- `request` — доступ, запрос, нужна помощь, не работает, клиент
- `hr` — отпуск, больничный, сотрудник, рабочее место, график
- `equipment` — ремонт, не включается, сломался, замена, оборудование
- `monitoring` — cpu usage, disk usage, healthcheck, uptime, мониторинг
- `info` — дайджест, инструкция, созвон, демо, техническое задание
- `spam` — выиграли приз, нажмите на ссылку, скидка 90%, верификация аккаунта
- `finance` — счёт, оплата, акт, бухгалтер
- `uncategorized` — сюда падает всё что не подошло

Сама классификация сделана простым способом: для каждой категории есть список ключевых слов, идём по категориям в порядке приоритета, ищем слова в теме и теле через `if word in text`.

```python
# порядок проверки в Classifier
category_order = ["monitoring", "incident", "spam", "hr", "equipment", "request", "finance", "info"]
```

Если ни одно слово не нашлось — `uncategorized`. Приоритет важен: письмо про «ошибка 500, прошу доступ» должно уйти в incident, а не в request. Пример правил:

```python
rules["incident"] = ["ошибка 500", "сбой", "не отвечает","критический инцидент", "не могу войти"]
rules["spam"] = ["выиграли приз", "нажмите на ссылку", "скидка 90%", "верификация аккаунта"]
```

Порядок выставили так: сначала monitoring и incident (критичное, нельзя пропустить), потом spam (тоже не должен попасть в обычные), потом hr и equipment, потом request и finance, потом info (самая общая категория, забирает остатки).

Новые типы писем: если завтра появится что-то новое, оно попадёт в uncategorized. Чтобы добавить категорию, надо дописать ключевые слова в `Classifier.make_rules()` и добавить имя папки в `Processor.categories`. Логику переписывать не надо.

После прогона на реальном наборе получилось так:

```text

uncategorized: 20 писем 18.3%
incident: 12 писем 11.0%
hr: 11 писем 10.1%
info: 10 писем 9.2%
finance: 7 писем 6.4%
equipment: 7 писем 6.4%

Итого 109 писем
```

## Основные файлы

```
1. mail_system.py (главный файл) # Message, Reader, Classifier, Processor
2. main.py	# точка входа
3. run.sh
4. test_mail_system.py
5. requirements.txt	# pytest
6. .gitignore
7. inbox/     # 109 писем (105 txt, 1 json, 1 jpeg, 1 bin, 1 без расширения)
8. processed/ # создаётся при запуске
```

Код разбили на 4 класса. Message хранит всё про одно письмо: путь к файлу, имя отправителя, тему, расширение, категорию и ошибку. 

Reader читает файлы из inbox/. Он умеет читать обычный текст, json, файл без расширения. Также он отдельно обрабатывает пустые и бинарные файлы.

Classifier отвечает за выбор категории. Он хранит правила и проверяет письмо по ключевым словам.

Processor связывает всё вместе: создаёт папки, берёт письма у Reader, отдаёт их в Classifier, копирует файлы в processed/ и считает статистику.

## Запуск

```bash
cd hackaton_tp
pip install -r requirements.txt
chmod +x run.sh
./run.sh
```

Можно напрямую `python3 main.py`

`run.sh` проверяет есть ли папка inbox, считает файлы, запускает `python3 main.py`, потом обходит `processed/` и выводит раскладку:

```bash
if [ ! -d "inbox" ]; then echo 'Ошибка: папка inbox не найдена'; exit 1; fi
for dir in processed/*/; do echo " $dir: ..."; done
```

Всего сделали 8 тестов - 6 обычных, 1 параметризованный и 1 интеграционный.

- `test_read_normaltext` — обычное текстовое письмо с Subject/From/телом
- `test_read_emptyfile` — пустой файл, должен вернуть `msg.error = 'Пустой файл'`
- `test_read_binary` — .png файл, должен определиться как бинарный
- `test_read_missfolders` — если папки нет, возвращается пустой список

- `test_read_jsonmail` — JSON-письмо, проверяем что subject/sender вытащились

## Ошибки и логи

Лог в `processing.log`, стандартный `logging.basicConfig`. Что может пойти не так:

- нет папки inbox — программа завершается, скрипт даёт `exit 1`
- бинарный файл — проверяем расширение по списку, не читаем, warning в лог
- пустой файл — проверяем после чтения что `text.strip()` пустой, warning
- битый JSON — `try/except json.JSONDecodeError`, пишем содержимое как строку, warning
- ошибка чтения файла — `try/except`, error в лог, письмо пропускается
- ошибка копирования — `try/except` вокруг `shutil.copy2()`, error в лог
- нет подходящей категории — `uncategorized`, система продолжает

На реальном прогоне сработали: пустой файл (mail_0103.txt), бинарный .bin (mail_0104.bin), битый JSON (mail_0105.json), бинарный .jpeg (mail_0109.jpeg). Всё отработало без падений.

```text
inbox/ - Reader - Message - Classifier - Processor - processed/ + log + статистика
```
